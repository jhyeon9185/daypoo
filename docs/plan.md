# 배포 환경 로그인 429(Too Many Requests) 에러 해결 계획

## 1. 개요
배포 환경의 로그인 페이지에서 `429 Too Many Requests` 오류가 발생하는 문제를 진단하고, 보안과 사용성을 고려하여 Rate Limit 설정을 최적화합니다.

## 2. 원인 분석
- **엄격한 제한**: 현재 로그인 API는 **5분당 5회**로 제한되어 있어, 비밀번호 오입력이나 환경적 요인(공유 IP 등)으로 인해 정당한 사용자가 차단될 가능성이 높음.
- **중복 호출 가능성**: 프론트엔드에서 로그인 시도 시 의도치 않게 중복 API 요청을 보낼 경우 즉시 임계치에 도달함.
- **기타 검증 API**: 이메일/닉네임 중복 확인 API도 타이핑 시마다 호출될 경우 **1분당 10회** 제한에 걸릴 수 있음.

## 3. 개선 방안

### A. 백엔드 (Rate Limit 완화 및 최적화)
- **로그인 제한 완화**: 
  - 기존: 5회 / 300초 (5분)
  - 변경: **10회 / 300초** 또는 **5회 / 60초**로 조정하여 사용자 편의성 증대.
- **중복 확인 제한 완화**:
  - 기존: 10회 / 60초
  - 변경: **20회 / 60초** (프론트엔드 디바운스가 미흡할 경우를 대비).

### B. 프론트엔드 (요청 최적화 가이드)
- **로그인 버튼 비활성화**: 요청 중에는 버튼을 비활성화하여 중복 클릭 방지.
- **디바운스 검합**: 중복 확인 API 호출 시 최소 500ms 이상의 디바운스 적용 확인.

## 4. 작업 단계
1. **상태 확인**: 현재 `AuthController.java`의 `@RateLimit` 설정값 확인 (완료).
2. **백엔드 수정**: `AuthController.java`의 속성값 조정.
3. **배포 및 테스트**: 배포 환경에서 429 에러 재발 여부 모니터링.
